# Output Contract

Every skill emits artifacts into `runs/<slug>/` (fresh run outputs) or `$OBSIDIAN_VAULT/TeamX/**` (long-term vault) following a stable shape so subagents can read each other's work without guesswork. All schemas below are **authoritative** - skills that do not conform must be fixed, not worked around.

`<slug>` format: `YYYY-MM-DD_<topic-kebab>`, generated by `orchestrate-multi-ai` at the start of a run.

## 1. Raw source answer - `runs/<slug>/raw/<source>.json`

One file per AI site or tool, written by a `browser-operator` or `research-scout` invocation. Even on failure you MUST write this file, with `error` populated.

```json
{
  "source": "gpt",
  "model": "gpt-4o",
  "question": "<the exact prompt sent to the site>",
  "answer_md": "<markdown body of the assistant reply, or null on error>",
  "url": "https://chatgpt.com/c/<conversation-id>",
  "ts": "2026-04-13T12:34:56Z",
  "citations": [
    { "title": "...", "url": "...", "snippet": "..." }
  ],
  "error": null
}
```

- `source` - stable short name: `gpt`, `gemini`, `notebooklm`, `grok`, `x-scan`, `tavily-search`, etc.
- `citations` - optional array; empty array is fine, but include it when the site surfaces sources (NotebookLM, Gemini, Grok).
- On failure, set `answer_md: null` and `error: { stage, message, screenshot_path? }`.

Tavily outputs live in their own subfolder. Search results use `runs/<slug>/raw/tavily/search-<n>.json` (one file per query variation, with `<n>` starting at 1). Extract and crawl outputs use `raw/tavily/extract/<domain>.json` and `raw/tavily/crawl/<domain>.json`. Each wraps the raw Tavily response under a `result` key and adds the same `source` / `ts` / `error` fields.

The Tavily lane also writes `runs/<slug>/raw/tavily/status.json` as a control artifact. This file is the authoritative completion sentinel for the Tavily lane; verifiers must not extract claims from it.

```json
{
  "source": "tavily-status",
  "mode": "background|inline-fallback",
  "ts": "2026-04-13T12:36:00Z",
  "searches_written": 5,
  "extracts_written": 2,
  "completed": true,
  "error": null
}
```

- `mode` distinguishes a background `research-scout` run from a main-session inline fallback.
- `searches_written` / `extracts_written` count files written under `raw/tavily/`, including error stubs.
- On lane-level failure, still write `status.json` with `completed: false` and `error: {stage, message}`.

## 2. Evidence pack - `runs/<slug>/evidence.json`

Produced by `evidence-verifier` (via the `build-evidence-pack` skill). Normalizes the raw files into atomic claims with provenance.

```json
{
  "slug": "2026-04-13_moe-architectures",
  "question": "<user's original question>",
  "created": "2026-04-13T12:40:00Z",
  "claims": [
    {
      "id": "c01",
      "text": "<single-sentence factual claim>",
      "sources": ["gpt", "gemini", "tavily:arxiv.org/abs/..."],
      "confidence": "high|medium|low",
      "notes": "<optional reviewer note>"
    }
  ],
  "contradictions": [
    { "claim_ids": ["c03", "c07"], "note": "<how they disagree>" }
  ],
  "open_questions": [
    "<things no source addressed>"
  ]
}
```

## 3. Final brief - `runs/<slug>/brief.md`

Produced by `synthesis-editor`. Reads `evidence.json` only (not raw). Max 800 words unless the user asked otherwise. Frontmatter is required and must match exactly so Obsidian search and Bases queries keep working.

```markdown
---
title: "<Human Title>"
slug: 2026-04-13_moe-architectures
created: 2026-04-13T12:45:00Z
tags: [teamx, <topic-tag>]
sources: [gpt, gemini, notebooklm, grok, tavily]
question: "<user's original question>"
---

## TL;DR
<2-3 sentences>

## Key findings
- ...

## Disagreements
- ...

## Open questions
- ...

## Sources
- [gpt] runs/<slug>/raw/gpt.json
- [gemini] runs/<slug>/raw/gemini.json
- ...
```

`archive-curator` later copies this file unchanged (aside from injecting `teamx` into `tags` if missing) to `$OBSIDIAN_VAULT/TeamX/Runs/<slug>/brief.md`. The frontmatter is what `Index/runs.base` reads - if you change fields here, update `.claude/skills/refresh-indexes/SKILL.md` too.

## 4. Daily log entry - `$OBSIDIAN_VAULT/TeamX/Daily/<YYYY-MM-DD>.md`

Produced by `journal-curator` (via the `log-day` skill), appended one **session block** per invocation. The file is append-only; prior blocks are never rewritten. Frontmatter is written once when the file is created and may only be mutated to extend `runs:` with new wikilinks (dedup required).

```markdown
---
date: 2026-04-13
tags: [teamx, daily]
runs:
  - "[[Runs/2026-04-13_moe-architectures/brief|MoE]]"
  - "[[Runs/2026-04-13_us-iran-conflict-latest/brief|US-Iran]]"
---

# 2026-04-13

## 14:32 - <session topic>
- **做了什么**:
  - ...
  - ...
- **关键决策/学到的**:
  - ...
- **产物**:
  - [[Runs/2026-04-13_moe-architectures/brief]]
  - [[Knowledge/moe/2026-04-13_moe-architectures-c03]]
- **未决/待续**:
  - ...

## 16:05 - <next session topic>
- ...
```

Rules:
- `date:` equals the filename date. Never back-date or re-date a file.
- `tags:` must contain `teamx` and `daily`. Other tags optional.
- Session blocks are H2 headings `## HH:MM - <topic>`, always in local time, monotonically increasing.
- Sub-sections with empty bullet arrays are omitted entirely.
- `runs:` and `artifacts` use wikilinks (vault-relative, no `.md` extension). Prefer aliasing with `|`.

## 5. Knowledge atom - `$OBSIDIAN_VAULT/TeamX/Knowledge/<topic>/<slug>-<claim-id>.md`

Produced by `knowledge-curator` (via the `promote-claim` skill) on an explicit user request. One file per promoted claim. The atom is an evergreen note; its body may be hand-edited by the user later - only `knowledge-curator` and the user should ever write it.

```markdown
---
title: <claim rewritten as a short phrase, max 12 words>
type: knowledge-atom
tags: [teamx, <topic>, <other-tags-from-brief>]
source_run: "[[Runs/2026-04-13_moe-architectures/brief]]"
source_claim_id: c03
confidence: high
created: 2026-04-13
---

> <claim text, verbatim from evidence.json>

## 上下文
<2-4 sentences extracted from brief.md that surround the claim>

## 来源
- [[Runs/2026-04-13_moe-architectures/raw/gpt.json|gpt]]
- [[Runs/2026-04-13_moe-architectures/raw/gemini.json|gemini]]

## 相关
- [[Knowledge/moe/...]]
- [[Knowledge/quantization/...]]
```

Rules:
- `type: knowledge-atom` is mandatory - `Index/knowledge.base` filters on it.
- `source_run` is a wikilink, not a raw path. `source_claim_id` is the verbatim `id` from `evidence.json`.
- `tags:` must contain `teamx`; the first non-teamx tag is the `<topic>` folder name.
- Source rendering is strict: plain source names become raw-file wikilinks like `[[Runs/<slug>/raw/grok.json|grok]]`. Only source strings containing `:` may be split into `<label>:<value>` and rendered as an external link; if `<value>` cannot be reliably normalized into a URL, emit the original source string as plain text instead of inventing a link.
- `## 相关` is populated by grepping existing atoms with overlapping tags. Max 5 links; if none, emit a single `- (none)` line so the section exists.
- On re-promotion (same `slug` + `claim_id`), the body may be overwritten but the `## 相关` section must be preserved if the user has hand-curated it.

## 6. Index views - `$OBSIDIAN_VAULT/TeamX/Index/*.base`

Produced by `knowledge-curator` (via the `refresh-indexes` skill). Three fixed files: `runs.base`, `knowledge.base`, `daily.base`. These are **derived artifacts** - they may be regenerated wholesale any time. Do not hand-edit them; see `.claude/skills/refresh-indexes/SKILL.md` for the canonical YAML bodies.

## Failure discipline

A skill that cannot produce its artifact MUST still write a stub file with `error` populated so the downstream verifier can see it tried. Silent skips break the evidence chain. Curators that fail to write into the vault must still print a final message explaining which lane failed and why - never swallow a vault write error.
